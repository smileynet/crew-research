## Purpose

Zod is a TypeScript-first schema validation library. You define declarative schemas, parse untrusted data against them, and get back strongly-typed, validated results with static type inference via z.infer<>. Zero runtime dependencies, works in Node.js + all modern browsers. The 2kb gzipped core makes it viable in bundle-sensitive contexts.

### Layered design

1. Core (v4/core) — schema definitions, runtime validation logic (_zod.run), error types, JSON Schema interop. Framework-agnostic internals.
2. Classic (v4/classic) — chainable OOP API that most users see (z.string().min(5).email()). Wraps core with method chaining, coercion helpers, and parse utilities.
3. Mini (v4/mini) — same core engine, but exposed as standalone functions (z.string(), z.pipe(...)) for better tree-shaking. No prototype methods.

### Key design patterns

- **Trait-based inheritance** — $constructor() creates schema "classes" using a trait system (_zod.traits Set) rather than classical inheritance. Symbol.hasInstance overrides enable instanceof checks against traits.
- **Single-pass parsing** — schemas implement _zod.run(payload, ctx) which mutates a ParsePayload (value + issues array). Synchronous by default; returns a Promise only when async refinements are involved.
- **Immutable public API** — every method returns a new schema instance (e.g., .optional() wraps in $ZodOptional).
- **Conditional exports** — the @zod/source condition lets development tooling (tsx, vitest) resolve directly to .ts source without building.
- **util.defineLazy()** — deferred property initialization to break circular dependencies between schema types.

## Complexity

- **Large** — the core schemas.ts alone is ~148KB / ~4000 lines. The classic wrapper adds another 97KB. The v3 compat layer (types.ts) is 160KB.
- **Deep type-level programming** — heavy use of conditional types, mapped types, infer, template literal types. The TypeScript complexity is at least as significant as the runtime complexity.
- **40+ locales** — each with ~4–11KB of error message templates.
- **Dual API surface** — classic (OOP) and mini (functional) must stay in sync feature-wise.
- **v3 compat maintained** — the entire Zod 3 codebase lives alongside v4, exported as zod/v3.

## Patterns

| Pattern | How it's used |
|---------|--------------|
| Monomorphic hot paths | Parameter reassignment explicitly allowed (biome noParameterAssign: off); performance-critical parse loops avoid allocations |
| /*@__PURE__*/ / /*@__NO_SIDE_EFFECTS__*/ | Extensive tree-shaking annotations |
| Discriminated unions | Schema types use def.type string literal discriminant |
| Standard Schema v1 | Implements the emerging StandardSchemaV1 interface for interop |
| JSON Schema bidirectional | Both toJsonSchema() and fromJsonSchema() |
| Registry pattern | Named schema registries for metadata/doc attachment |
| JIT union matching | Discriminated unions get a fast-path compiled matcher |

## Coding Style

- **TypeScript throughout** — no JavaScript source files
- **ES modules** — "type": "module" everywhere; CJS output generated by build only
- **Biome** for linting/formatting — 2-space indent, 120 char line width, ES5 trailing commas
- **Vitest** for tests with full typecheck enabled (checker: "tsc")
- **Naming:** $-prefixed types/classes for internal/low-level constructs ($ZodString, $ZodType). Public API uses unprefixed names (z.string(), ZodError).
- **No JSDoc clutter** — minimal comments; types speak for themselves
- **Performance-first tradeoffs** — mutable payload objects during parse, parameter reassignment, manual loop unrolling in hot paths
- **Husky hooks** — pre-commit runs biome (format + lint), pre-push runs full vitest suite
- **play.ts** at workspace root for quick experimentation via pnpm dev:play
