> ## Project Overview: Zod

### Purpose

This is Zod (v4.4.3) — the dominant TypeScript-first schema validation library. It lets you declare schemas, parse/validate data at runtime, and infer static TypeScript types from those schemas. Zero external dependencies, ~2KB gzipped core. Created by Colin McDonnell, MIT licensed, published to npm and JSR.

### Architecture

pnpm monorepo with these workspace packages:

| Package | Role |
|---------|------|
| packages/zod | The library itself |
| packages/bench | Runtime benchmarks (mitata, tinybench) |
| packages/docs | Next.js documentation site (zod.dev) |
| packages/docs-v3 | Legacy v3 documentation (static HTML) |
| packages/integration | Integration test fixtures |
| packages/resolution | Module resolution correctness tests (arethetypeswrong) |
| packages/tsc | TypeScript compiler performance tests |
| packages/treeshake | Bundle size analysis (rollup tree-shaking) |

Internal layering of packages/zod/src:

v4/core/          ← foundation: schemas, checks, parse, errors, util, JSON Schema, registries
v4/classic/       ← full API: extends core with coercion, compat, ISO types, fromJSONSchema
v4/mini/          ← tree-shakeable minimal variant (function-based, no method chaining)
v4/locales/       ← i18n error messages (40+ languages)
v3/               ← legacy v3 compatibility layer (monolithic types.ts, still ships for migration)


Multiple export paths: zod, zod/mini, zod/v3, zod/v4/core, zod/locales, zod/v4/locales/*.

### Complexity

High. Key indicators:

- **v4/core/schemas.ts** (148KB) — all core schema type definitions and internals
- **v4/classic/schemas.ts** (97KB) — classic API factory functions
- **v3/types.ts** (160KB) — monolithic legacy implementation
- **Custom OOP system** — $constructor trait-based initialization instead of class inheritance. Schemas carry a _zod internal object with traits, def, run, parse, and JIT compilation hooks.
- **JIT compilation** — schemas compile into optimized parse functions (with jitless escape hatch for CSP-restricted environments)
- **Heavy type-level programming** — conditional types, mapped types, template literal types for full input/output inference
- **Dual CJS/ESM** — built via zshy with custom condition @zod/source for source-mode development

### Key Patterns

- **Immutable API** — every method returns a new schema instance
- **Trait system** — $constructor(name, initializer) replaces class hierarchies; Symbol.hasInstance overridden for instanceof checks against trait sets
- **@zod/source condition** — development runs directly against .ts source files (no build step during dev), production uses compiled outputs
- **Performance-first** — @__PURE__ / @__NO_SIDE_EFFECTS__ annotations, sideEffects: false, parameter reassignment explicitly allowed, Object.defineProperty for non-enumerable internals
- **Standard Schema** — implements the StandardSchemaV1 interop interface
- **Global config** — globalThis.__zod_globalConfig shared across CJS/ESM boundaries

### Users

TypeScript/JavaScript developers validating data at trust boundaries — API endpoints, form submissions, config files, deserialization. The ecosystem is enormous: React Hook Form, tRPC, Drizzle, Hono, hundreds of integrations. Ships with llms.txt and an MCP server endpoint for AI tooling.

- **Node 22+**, TypeScript ~5.5.4, ES modules throughout
- **Biome** for formatting + linting (not ESLint/Prettier) — 2-space indent, 120-char line width, trailing commas ES5 style
- **any is allowed** and considered a feature (biome rule noExplicitAny: off)
- **Parameter reassignment allowed** for performance (biome rule noParameterAssign: off)
- **Vitest** with typecheck enabled — tests co-located in tests/ subdirectories next to source
- **Husky** — pre-commit runs biome via lint-staged; pre-push runs the full test suite
- **No unused imports** (error-level), unused variables (warn-level)
