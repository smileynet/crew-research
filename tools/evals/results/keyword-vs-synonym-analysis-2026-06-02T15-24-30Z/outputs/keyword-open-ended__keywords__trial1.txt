> ## Comprehensive Project Analysis: Zod

### Jobs-to-be-Done

Zod solves runtime validation with compile-time type inference — the problem of having TypeScript types that don't enforce anything at trust boundaries (API responses, user input, config files). Users define a schema once and get both the validator and the TypeScript type derived from it.

The v4 rewrite specifically addresses:
- **Bundle size**: tree-shakeable architecture with a zod/mini entrypoint for size-sensitive apps
- **Performance**: JIT-compiled parse functions, parameter reassignment for hot paths, @__NO_SIDE_EFFECTS__ annotations throughout
- **Ecosystem maturity**: built-in JSON Schema conversion (both directions), Standard Schema compliance, i18n (40+ locales), and a compat shim for v3 users

### Architecture & Key Decisions

Layered module structure:
src/index.ts          → re-exports v4/classic (the "batteries included" default)
src/v4/core/          → engine: schemas, parse logic, errors, checks, JSON schema
src/v4/classic/       → user-facing API (wraps core with ergonomic factories + english locale auto-configured)
src/v4/mini/          → minimal API for bundle-size-conscious consumers
src/v3/               → backward-compat shim (160KB types.ts monolith, legacy API surface)


Custom class system via $constructor: Rather than standard ES classes, Zod uses a factory function ($constructor) that builds constructors with trait-based composition (traits: Set). This enables:
- instanceof checks via Symbol.hasInstance that test trait membership
- Multiple inheritance without class diamond problems
- Lazy prototype binding for schema methods

Parse pipeline is payload-based: A mutable ParsePayload { value, issues, aborted?, fallback? } flows through the schema graph. This avoids allocating result wrappers per node and enables abort-on-first-error semantics.

Custom condition @zod/source: Development runs against raw .ts sources via Node's --conditions flag. Tests, bench, and play.ts all resolve to source without a build step. The published artifact is pre-compiled by zshy (a bespoke build tool).

### Tradeoffs

| Decision | Benefit | Cost |
|---|---|---|
| Single-package multi-entrypoint (zod, zod/mini, zod/v3, zod/v4/core) | One npm install, fine-grained imports | Complex exports map, resolution testing needed (packages/resolution) |
| Mutable payload parse | Zero allocation in the happy path | Subtle bugs when transforms/catches interact (see wiki/optionality.md) |
| noParameterAssign: off in linter | Measurable perf gain in _parse loops | Harder to reason about data flow |
| Three-file version lock (package.json + jsr.json + versions.ts) | Catches accidental publishes via check:semver | Manual coordination required; any mismatch blocks commit |
| v3 compat layer shipped inside v4 | Smooth migration, one dependency | Carries 160KB of dead weight for new projects |

### Non-Obvious Gotchas

1. Version bump = irreversible release. Pushing a version change to main triggers npm publish + JSR publish + GitHub Release. There's no draft step. The prepublishOnly + pre-commit hooks gate this, but a force-push bypasses them.

2. Console output in tests is a hard error. scripts/fail-on-console.ts replaces all console.* methods with throwers. Any console.log left in test code or accidentally triggered production code will fail the suite.

3. Untracked files block push. Both pre-commit and pre-push reject if git ls-files --others --exclude-standard shows anything. You must either commit or .gitignore scratch files before pushing.

4. TypeScript pinned to 5.5. The .nvmrc says Node 22, and TS is ~5.5.4. CI tests against both 5.5 and latest, but local dev uses the pinned version. Type-level tricks in schemas.ts may rely on 5.5 behavior.

5. optin / optout / fallback interaction. The wiki/optionality.md documents a hard-won understanding of how optional/catch/default/transform interact. This is the most complex part of the runtime — any change to schema composition involving these signals needs careful reasoning about the three flags.

6. Circular dependency check. madge --circular runs in CI excluding v4/core and v3. The core module has known/accepted internal cycles; adding cross-module imports requires verifying they don't create new circles.

7. Canary publishes on every push to main. If the version doesn't change, the release workflow still publishes a canary (x.y.z-canary.TIMESTAMP). This means every merge to main produces an npm artifact.

### Prior Art & Positioning

- **io-ts** — Zod's predecessor in the TS validation space; more FP-oriented with codec/decoder separation. Zod chose class-based ergonomics instead.
- **Yup** — Zod explicitly targets its API convenience while delivering stronger types.
- **Valibot** — Direct competitor on bundle size. The packages/treeshake workspace exists specifically to compare Zod vs Valibot tree-shake outcomes. Zod counters with the mini entrypoint.
- **ArkType** — Listed as a devDep (benchmarking). Competes on type inference speed.
- **Standard Schema** — Zod implements StandardSchemaV1, a cross-library interop contract (see standard-schema.ts).

### Audience

1. Application developers using the default zod import — want DX, error messages, and comprehensive validators.
2. Library authors using zod/mini or zod/v4/core — want small bundles and programmatic schema construction.
3. Migrating v3 users using zod/v3 — want to upgrade gradually.
4. Ecosystem tools (ORMs, form libs, tRPC) — consume the Standard Schema interface or the JSON Schema output.

### Conventions

- **ES modules everywhere** ("type": "module"). No CJS source.
- **Biome** for formatting (2-space indent, 120 line width) and linting. Prettier only for markdown.
- **Vitest** with typecheck enabled — tests validate both runtime behavior AND type inference.
- **No console statements** in tests or production code.
- **play.ts** for experimentation (pnpm dev:play); tests for everything permanent.
- **zshy** as the build tool (generates CJS/ESM/dts from the single TS source).
- **Husky hooks**: pre-commit runs lint-staged + semver check; pre-push runs the full test suite.
- **pnpm workspaces** with hoistWorkspacePackages: false (strict isolation).
- **Performance annotations**: @__NO_SIDE_EFFECTS__, @__PURE__ on constructors and factory functions for dead-code elimination.
