# World-Building Specification

Define the rules before writing in the world. Explicit boundaries prevent drift across scenes/stories.

## Premise Specification Checklist

Every world/system/conceit must explicitly answer:

1. **What does it DO?** — specific capabilities with examples
2. **What does it NOT do?** — explicit boundaries (prevents the most common generation errors)
3. **What happens when it fails?** — specific observable behavior, not vague
4. **How do people respond to failure?** — collective behavior patterns
5. **What's the social model?** — ubiquitous vs. elite, or spectrum
6. **What vocabulary is BANNED?** — prevents collection-level tics across scenes

## Why Explicit Boundaries Matter

Leaving questions 2, 3, 4, and 6 implicit produces inconsistent results. Explicit corrections with examples produce 100% compliance. The most common failure: the system does something the author didn't intend because "what it doesn't do" was never stated.

## Template

```markdown
# [World/System Name]

## Capabilities
- [What it does, with specific example]
- [What it does, with specific example]

## Boundaries (what it CANNOT do)
- [Explicit limitation]
- [Explicit limitation]

## Failure Mode
- Observable behavior: [what happens when it breaks]
- Social response: [how people react]

## Social Model
- [Who has access, how it's distributed, what the class dynamics are]

## Banned Vocabulary
- [Words/phrases that must not be used to describe this system]
- [Reason: prevents repetitive or inaccurate description]
```

## Rules

- Answer ALL 6 questions before writing any scene in this world
- Boundaries are more important than capabilities (capabilities emerge; boundaries prevent errors)
- Update the spec when you discover a new boundary during writing
- Each story/scene in this world must be checked against the spec for consistency
